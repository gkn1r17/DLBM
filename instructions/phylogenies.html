Step 1: Producing phylogeny raw data during run

Add MUTANT_TIMESTEPS_DAY timestepsYearly.csv to the command line when running your simulation.
This will export and flush the information needed to build a phylogeny to the subfolder phylogenies every year (you can substitute other timestep csv files with other intervals but it doesn't affect results and this seems to work fine for performance).
Phylogenies are saved (and exported in a Java compressed format) with information [parent, number_of_children, time] for each location and each mutation.
Each mutation spawns a new child lineage.
Children ID and location ID can easily be deduced as the ID's of new lineages increment by 1 each time, which enables a complete phylogeny to be constructed offline (See below) even though it is stored in a compressed format during the simulation
This is done for efficiency reasons because, as phylogenies are only traced back for surviving lineages, most new lineages produced will never need to be reconstructed in this way.
Birth time is also calculated offline to the granularity specified in e.g. REPORT_TIMESTEPS_DAY report10Days.csv (this example = 10 days).

Step 2: Phylogeny offline collation

The Java method responsible for offline phylogeny collation is inputOutput.CollatePhylogenies.main(...)
Will take several hours for K=1000 and around 6000 grid boxes.
Can be stopped early, if so will have collated up up to a particular time in the past (i.e. not to beginning of simulation).
This example is for collating at year 10000, can also collate starting at earlier years, which will also take less time and may be possible with higher K.
These instructions are for Iridis/(SLURM based cluster) and will need to be adapted for local Linux/Windows environment. runPhylo.slurm will also need to be adapted.
1. Make sure runPhylo.slurm and collatePhylo.jar are in the run folder
2. Run collation with the command below. Modify as required:
- "3650000" will trace back the phylogeny for every lineage alive at day 3650000 (year 10000) and can be changed to earlier year to trace back phylogeny from earlier year. - "280" is just a parallelisation parameter optimised for the number of cores available. For more powerful servers shouldn't need changing.
- "FILE_LOAD phylo3/mut_T11-04-2026*18-34-53" should be the path to the run for which phylogeny is being collated and complete file name including timestamp with space replaced with "*"
- Other parameters should match parameters which with run was produced.
3. Output printed to standard out (temp soluting should probably be fixed at some point).
- On Iridis the destination of standard out can be changed in runPhylo.slurm by addeding e.g. #SBATCH --output=my_phylo.out, otherwise will output to slurm-[job number].out
4. When the run has finished output will need to processed into .mat object as follows:
- convert to csv
- remove lines except those starting with "phylo"
- run collatePhylo2.m 

                
                     sbatch runPhylo.slurm 280 3650000 NUMNODES 2 SETTINGS testSettings/testMutation FILE_OUT mutPhylo/mut DURATION_DAY 3650000 DISP_SCALER 1 SEED 0 MUTANT_TIMESTEPS_DAY timestepsYearly.csv SAVE_TIMESTEPS_DAY timestepsYearly.csv REPORT_TIMESTEPS_DAY report10Days.csv FILE_LOAD phylo3/mut_T11-04-2026*18-34-53 LOAD_DAY 3650000 TEMP_FILE none CLUST_FILE clusters6386x.csv
